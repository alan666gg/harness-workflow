# Harness Commander

## 概述

这个 skill 专门给主线程用。

它不强调“自己把所有代码都写完”，而是强调：

- 看清总目标
- 拆出合理任务
- 控制 scope
- 维护计划
- 统一验收

如果 `harness-dev-workflow` 更像“怎么把任务做完”，
那 `harness-commander` 更像“怎么把一组任务组织起来并收口”。

## 什么时候用

- “帮我规划整个开发计划”
- “把这个大需求拆成几条任务”
- “现在子线程都跑完了，帮我复盘”
- “需要决定下一步先做哪条”
- “需要统一验收多个 tranche”

## 不该用来做什么

下面这些不应该是它的主职责：

- 深入某一个模块连续写实现代码
- 在没有 task 边界的情况下随手改很多文件
- 把 commander 变成大而全的执行线程

如果用户没有明确要求分工，Commander 也可以暂时兼任实现，但要先把任务边界写清楚。

## 核心职责

1. 维护总计划
2. 把大目标拆成可执行任务
3. 为每个任务补 handshake
4. 判断任务依赖关系
5. 控制线程边界，避免重叠修改
6. 收集验证结果
7. 生成 follow-up

## Commander 的标准动作

### 1. 先读总入口

优先读取：

1. `AGENTS.md`
2. `PLAN.md`
3. `docs/harness/README.md`
4. `docs/harness/PROJECT_MAP.md`
5. `docs/harness/THREAD_PROTOCOL.md`

### 2. 定义当前主问题

先回答：

- 现在的目标是什么
- 最值钱的下一步是什么
- 哪个问题是 blocker
- 哪些问题只是 follow-up

### 3. 拆任务时按“边界而不是按话题”拆

好的任务拆分方式：

- 按模块边界拆
- 按写入文件边界拆
- 按先后依赖拆

不好的拆分方式：

- 多个线程同时改同一批核心文件
- 只按“感觉差不多”拆

### 4. 每条任务至少写清 5 件事

- 背景
- 目标
- ownership
- 验收标准
- 不做什么

### 5. 先做关键路径，不要沉迷并行

Commander 要先区分：

- **关键路径任务**
- **可并行 sidecar 任务**

如果下一步明显依赖某个结果，就应该优先推进关键路径，而不是为了并行而并行。

### 6. 验收时优先看真实问题

复盘或 review 时优先回答：

- 有没有新 blocker
- 有没有回归
- 当前真的完成了什么
- 还没完成什么

不要只写概述，不写风险。

## 输出建议

Commander 的输出优先按这个格式组织：

1. 当前项目进度摘要
2. 当前 blocker / 风险点
3. 下一批建议开启的任务
4. 每条任务可直接复制的 prompt

## 什么时候要新建 follow-up task

满足任一条件就建议新建：

- 当前任务刻意留了残余限制
- 发现新 blocker，但不适合在当前任务硬塞
- 新问题需要不同 ownership
- 已经形成清晰的下一 tranche

## 什么时候应该说“先停在这里”

当满足这些情况时，Commander 应该建议阶段收口：

- 当前 checkpoint 已经对产品有价值
- 再往下做会进入明显更大的系统改造
- 当前主瓶颈已经转移到别的 lane

## Commander 的完成标志

一轮 Commander 工作合格，至少要做到：

- 当前状态说清楚
- 下一步说清楚
- 任务边界说清楚
- 风险说清楚
- 结果写回仓库
