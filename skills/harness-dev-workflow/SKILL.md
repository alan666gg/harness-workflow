# Harness 开发工作流

## 概述

这个 skill 用来把 AI 开发从“靠聊天上下文推进”，变成“靠仓库文件推进”。

它融合两套方法：

- **文档驱动开发**：先澄清目标、验收标准、设计边界，再写代码
- **harness 文件化协作**：把计划、任务、握手、工作日志写回仓库，方便接力和复盘

默认支持两种模式：

- **单线程模式**：一个线程从计划到实现再到验证一路推进
- **多线程模式**：Commander 负责拆任务，Worker 按 task/handshake 执行

如果用户没有明确要求拆子线程，优先按**单线程模式**执行。

## 什么时候用

- “开始做一个复杂功能”
- “继续开发这个大需求”
- “把后端/前端/合约/测试环境一起收完”
- “需要可追踪的 AI 开发流程”
- “希望把开发过程沉淀到仓库，而不是散落在聊天里”

## 不必重流程触发的场景

下面这些情况可以走窄版流程，但仍然要有验证：

- 单点 bug 修复
- 文案微调
- 纯样式调整
- 小范围脚本修改
- 明确、局部、低风险的重构

窄版流程至少要保留：

1. 问题定义
2. 相关文件定位
3. 代码修改
4. 验证
5. 结果回写

## 核心原则

1. **仓库是事实来源**
   - 计划、任务、握手、日志写回仓库
   - 不把关键决策只留在聊天里

2. **文档先于代码**
   - 先明确目标、边界、验收，再进入实现
   - 但不为了文档而文档化

3. **上下文最小化**
   - 先读入口文件和当前 task
   - 只打开当前任务真正需要的代码和文档

4. **任务边界显式化**
   - 每个任务说明 scope、ownership、依赖、验收标准

5. **验证不是可选项**
   - 每次改动都要有对应验证
   - 大任务必须走完整验证梯度

6. **默认主动推进**
   - 没有明显风险时直接实现
   - 只有涉及高成本、不可逆、架构分叉或安全风险时才停下来确认

## 标准执行顺序

### 第 1 步：读取入口

优先读取这些文件：

1. `AGENTS.md`
2. `PLAN.md`
3. `docs/harness/README.md`
4. `docs/harness/PROJECT_MAP.md`
5. `docs/harness/THREAD_PROTOCOL.md`

如果仓库没有完整 harness 结构，就先按现有项目结构找最接近的计划和文档入口。

## 第 2 步：确定当前是全量流程还是窄版流程

快速判断：

- 如果是跨模块、跨阶段、需要部署/联调的任务：走**全量流程**
- 如果是单点修复：走**窄版流程**

## 第 3 步：先写清楚任务合同

无论单线程还是多线程，都至少明确这 4 件事：

- 当前目标是什么
- 成功标准是什么
- 哪些文件允许改
- 用什么方式验证

如果仓库已经有 harness 目录，优先使用：

- `Task`
- `Handshake`
- `Worklog`

字段规范见：

- `references/file-contracts.md`

## 第 4 步：文档先行，但只补必要文档

优先补这些内容：

- 用户目标 / 业务语义
- 架构边界变化
- 测试与验收口径

如果任务较大，直接套用文档驱动开发的 6 步骨架：

1. User Story
2. 技术方案
3. 测试方案
4. 实现
5. E2E / 环境验证
6. 提交与发布

如果需求只是局部实现，不必强行补长篇设计文档；至少把：

- 目标
- 约束
- 验证口径

写进 task / handshake。

## 第 5 步：实现时按 tranche 推进

复杂任务不要一次把所有东西混着改。

推荐按 tranche 推进：

1. 先打通最小真实链路
2. 再补真实性和边界
3. 再补运维/读面/部署闭环

每个 tranche 都要能回答：

- 这轮做了什么
- 还没做什么
- 下一轮最自然的 follow-up 是什么

## 第 6 步：每轮都写 append-only worklog

worklog 只追加，不回写历史。

至少记录：

- 读了什么
- 改了什么
- 验了什么
- 还剩什么风险

## 第 7 步：按验证梯度收口

不要只跑单测就宣布完成。

标准梯度：

1. 局部测试
2. 模块测试
3. 本地 full flow / E2E
4. 测试环境 / 测试网验证
5. 部署后读面检查

详细检查口径见：

- `references/validation-and-release.md`

## 第 8 步：提交、推送、回写计划

完成后至少做这些：

- 更新 task 状态
- 更新 handshake 状态
- 追加 worklog
- 更新 PLAN 中的阶段状态
- commit / push

如果已经部署：

- 记录部署结果
- 记录测试环境验证结果
- 明确 residual limitations

## 单线程模式

如果不拆子线程，按下面方式执行：

1. 读取入口文档
2. 创建或更新当前 task / handshake / worklog
3. 补必要设计说明
4. 直接实现
5. 按验证梯度验证
6. 回写结果
7. 提交与推送

单线程模式依然要求：

- 有任务边界
- 有 worklog
- 有验证证据

## 多线程模式

如果用户明确要求拆子线程：

- Commander 负责：
  - 拆任务
  - 分 ownership
  - 维护总计划
  - 统一验收

- Worker 负责：
  - 只读当前 task/handshake 要求的最小上下文
  - 在指定边界内实现
  - 回传验证结果和 residual risks

## 结果输出的默认格式

每轮完成后，输出应尽量包含：

1. 当前项目进度摘要
2. blocker / 风险点
3. 下一步建议
4. 验证结果

如果是代码交付，优先说明：

- changed files
- validation commands
- 真实状态与残余限制

## 完成标志

一个任务可以视为真正完成，需要同时满足：

- 目标实现
- 验证完成
- 文档/任务状态已回写
- 残余限制已明确
- 后续任务已清楚

## 何时要停下来确认

只有这些情况才建议停下来确认：

- 会引入不可逆迁移
- 会影响生产数据或密钥
- 架构存在两条以上代价差异明显的路线
- 用户预算、时程或环境前提不明确

其他情况默认继续推进。
