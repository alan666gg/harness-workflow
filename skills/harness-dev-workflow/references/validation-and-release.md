# 验证与交付梯度

复杂任务建议按下面顺序收口。

## 1. 局部验证

目标：

- 先确认当前代码改动没直接坏

典型方式：

- 单元测试
- 模块测试
- lint / build
- 合约测试

## 2. 本地真实链路验证

目标：

- 不只看局部函数，而是看完整业务流

典型方式：

- 本地 full flow
- 本地 E2E
- 本地数据库 / 链 / 服务联动

## 3. 测试环境验证

目标：

- 确认部署后的真实环境仍然成立

典型方式：

- healthz
- API smoke
- 浏览器读面
- 测试网链路

## 4. 验证输出应包含什么

建议最少输出：

- validation commands
- 关键结果
- 失败点
- residual limitations

## 5. 部署前收口

部署前至少确认：

- worktree 干净或已知
- 任务状态已回写
- 测试口径清楚
- 风险点已说明

## 6. 部署后收口

部署后建议记录：

- commit hash
- deployment run
- health check
- staging / testnet 验证结果
- 仍未覆盖的边界

## 7. 默认发布纪律

- 小改动：本地验证通过即可推送
- 中改动：本地 + 测试环境验证
- 大改动：本地 full flow + 测试环境 + 测试网

## 8. 不应被当作“已完成”的情况

以下情况不能算真正完成：

- 只做了代码实现，没有验证
- 只在聊天里说明，没有写回仓库
- 只有 happy path，没有说明残余风险
- 测试环境失败，但没有明确 blocker 和下一步
